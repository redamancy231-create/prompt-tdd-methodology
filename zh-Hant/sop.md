# 對照實驗設計 SOP (Prompt Experiment Design SOP)

> **版本**: v1.0.0（從 prompt-tdd 專案提煉，泛化為通用協議）  
> **來源**: A2+A3 實驗的實戰方法論，經 17+ 輪異後端審查驗證  
> **生成模型**: DeepSeek-V4-Pro (via Claude Code CLI) · 2026-07-01

---

## §1 實驗前檢查清單 (CK1-CK6)

### CK1-CK3: Tier 1 硬門（缺一不可執行 Tier 1）

| 編號 | 檢查項 | 通過標準 |
|------|--------|---------|
| **CK1** | 假設可證偽 | H0/H1 明確陳述 + 拒絕域事前定義 + 統計檢驗方法指定 |
| **CK2** | 依變項可操作化 | DV 有明確定義 + 測量協議可複製 + 評分 rubric 凍結 |
| **CK3** | 樣本量充足 | 功效分析完成 + 最小興趣效應量宣告 + n ≥ min_n_per_arm |

### CK4-CK6: 條件觸發（依據實驗複雜度選擇性執行）

| 編號 | 檢查項 | 觸發條件 | 通過標準 |
|------|--------|---------|---------|
| **CK4** | INVENTORY 等價性 | 雙組 prompt 內容不同時 | 兩組 prompt 產出清單逐項核對 + hash 凍結，防 IV 汙染 |
| **CK5** | Train/Test 分離 | 需要迭代改進 prompt 時 | test 集在實驗結束前不被用於 prompt 選擇或參數調整 |
| **CK6** | 天花板探測 | DV 為 0/1 或比例指標時 | 嵌入反編造/極端測試用例評估檢測空間 |

## §2 實驗管線：Tier 0 → Tier 1

### 2.1 Tier 0: 描述性統計（工具鏈驗證）

**目的**: 在投入大樣本前驗證資料收集和評分管線可行。

| 步驟 | 操作 | 成功判據 |
|------|------|---------|
| 1. 小樣本收集 | n=4-8/臂，手動稽核每條原始輸出 | 無資料收集異常（截斷/超時/格式錯誤） |
| 2. 評分管線驗證 | 執行 score 腳本，檢查 manifest 完整性 | 評分腳本執行正確，無缺失分 |
| 3. 描述性統計 | `analyze_experiment.py --tier 0` | 方向性描述 + 效應量粗略估計 |
| 4. 工程門判決 | 判斷管線是否可進入 Tier 1 | PASS → 進入 Tier 1; FAIL → 修復後重跑 |

**Tier 0 不升級成熟度**。它只驗證"管線能用"，不提供推斷結論。

### 2.2 Tier 1: 推斷統計（假設檢驗）

**前置條件**: Tier 0 工程門 PASS + CK1-CK3 全部通過。

| 步驟 | 操作 | 關鍵約束 |
|------|------|---------|
| 1. 大樣本收集 | n ≥ 功效分析指定值 | raw_outputs/ 只增不覆蓋 |
| 2. 雙 LLM 異後端盲評 | 評分者 A（作者模型）+ 評分者 B（不同後端） | 評分者 B 不接觸 A 的評分、作者自評、材料來源 |
| 3. 解盲 + 合併 | 合併 A/B 評分者，對齊 case_id | 評分前不告知分組 |
| 4. 統計推斷 | `analyze_experiment.py --tier 1 --test-set test_set.json` | Wilcoxon + bootstrap CI + Cohen's κ |
| 5. 科學門判決 | 統計顯著 + 效應量 ≥ 最小興趣閾值 + 方向一致 | 三項全 PASS → 支援 H1; 任一 FAIL → 不支援 |

## §3 工程門 vs 科學門

對照實驗有兩道獨立門——混淆兩者是實驗設計最常見的錯誤。

### 工程門（Engineering Gate）

**判什麼**: 工具鏈是否可用？  
**通過標準**: 資料收集無異常 + 評分腳本執行正確 + manifest 完整性校驗通過  
**失敗後**: 修復工具鏈 → 重新收集（工程門失敗不影響科學結論——這是管道問題，不是假設問題）

### 科學門（Science Gate）

**判什麼**: 假設是否被支援？  
**通過標準**: 統計顯著 + 效應量 ≥ 最小興趣閾值 + 方向一致 + 評分者一致性達標（Cohen's κ ≥ 0.4 或 discordant 率 < 25%）（四項 **全部**）  
**失敗後**: 誠實記錄——陰性結果 ≠ 實驗失敗。A2/A3 科學門全部 FAIL，但仍然產出了有價值的陰性證據和方法論資產

> **關鍵教訓**：**工程門 PASS + 科學門 FAIL = 有效陰性證據**。成熟度標註為 [E-]（ceiling-limited）而非 [E]，但仍然是可發表的誠實結論。

## §4 預註冊鎖 (.lock)

### 4.1 為什麼需要

防止事後調整假設（HARKing——Hypothesizing After Results are Known）。

### 4.2 鎖內容

```yaml
hypothesis: "<H0/H1 声明>"
analysis_plan:
  test_method: "Wilcoxon signed-rank | sign test | ..."
  effect_size_threshold: <最小兴趣效应量>
  stopping_rule: "<停止规则>"
test_set_sha256: "<SHA256 hash of test_set.json>"
lock_date: "<ISO 8601>"
```

### 4.3 操作

1. 在實驗執行前，建立 `.lock` 檔案
2. `git commit` 後不可修改（任何修改都會產生新 commit hash，可追溯）
3. 分析腳本自動校驗 test_set SHA256——不匹配則 hard fail

## §5 INVENTORY 等價協議

### 5.1 問題

對照實驗的雙組 prompt 內容不同——但如何保證**資訊量等價**而非**格式效應**？

### 5.2 協議

1. 分別執行 A 和 B 組 prompt 在相同輸入上
2. 提取兩組 prompt 產出的**檢查項清單**（inventory）
3. 逐項核對兩組 inventory 是否覆蓋相同的檢查項
4. 對 inventory 做 hash 凍結
5. 如發現不等價 → 修正 prompt 直到等價 → 重新凍結

> **A2 教訓**：INVENTORY 等價性差異導致 IV（prompt 格式）與"資訊量差異"混淆——B 組實際包含了 A 組沒有的檢查維度。

## §6 成熟度標註體系

| 標籤 | 含義 | 升級條件 |
|------|------|---------|
| `[Sp]` | Specification — 設計草案 | 初始狀態 |
| `[E]` | Experiment — 實驗執行 | Tier 1 完成 + 科學門 PASS + 異後端審查閉合 |
| `[E-]` | Experiment — ceiling-limited | Tier 1 完成 + 科學門 FAIL（陰性或天花板） |
| `[F·1域]` | Framework — 單域整合 | [E] 結論寫回框架，標註任務域限制 |
| `[J]` | Journal — 學術發表 | 經同行評審 |

**約束**：單域實驗最多升級到 `[F·1域]`。跨域推廣需 ≥2 域實驗支援。

## §7 審查閉合協議

### 7.1 強制條件

- 異後端獨立審查 ≥1 輪後方可宣佈閉合
- 審查者不得與實驗執行者同後端
- 審查者須在裸環境中審閱（不接觸實驗執行記錄、作者自評、中間版本）

### 7.2 審查焦點

| 審查維度 | 檢查內容 |
|---------|---------|
| 設計合理 | CK1-CK6 全部通過？ |
| 執行正確 | 工程門 PASS？預註冊鎖有效？ |
| 分析正確 | 統計方法適當？結論不超出效應量範圍？ |
| 誠實記錄 | 陰性結果如實報告？侷限宣告充分？ |

## §8 常見失敗模式

### 8.1 天花板效應
**現象**: 模型接近完美 → 效應量不可檢測  
**檢測**: CK6 天花板探測（嵌入反編造/極端測試用例）  
**處理**: 誠實標註 [E-] ceiling-limited —— 這是有效結論，不是失敗

### 8.2 IV 汙染
**現象**: 對照組的 prompt 資訊量不等價 → 檢測到的是資訊量效應，不是格式效應  
**檢測**: CK4 INVENTORY 等價協議  
**處理**: 修正 prompt 直到等價 → 重新凍結

### 8.3 評分者分歧
**現象**: 兩個 LLM 評分者對同一 case 給出不同評分  
**檢測**: Cohen's κ < 0.4 或 discordant 率高  
**處理**: 採用保守評分策略（取較低分）；已知侷限——LLM-LLM 評分 ≠ 人類評分準確性

### 8.4 HARKing
**現象**: 看到結果後調整假設  
**檢測**: .lock 檔案的 git commit 時間戳在實驗執行之前  
**處理**: 預註冊鎖 + 科學門四項全 PASS 才能宣告"支援 H1"

### 8.5 DV 退化（A3v2 教訓）
**現象**: 實驗迭代中 DV 選擇逐步退化——從最初的構造效度關切，到後續版本中退化為"能拿到資料就行"  
**檢測**（A3v2 審查發現的四個設計門禁，Codex + Qwen 雙後端共識）：
1. **構造效度門**：DV 是否真正測量了假設指定的構念？（A3v2: accuracy 退化為0/1，丟失了路由決策的"品質"維度）
2. **格式×數量混淆門**：A/B 組 prompt 的格式差異是否與資訊量差異正交？（A3v2: 宣告式格式天然產生更多結構化欄位，格式和資訊量不可分）
3. **天花板逃脫門**：新的實驗設計是否創造了比上一輪更大的檢測空間？（A3v2: DV 退化後檢測空間反而更小）
4. **效應量可解釋門**：如果檢測到 Δ，它是什麼意思？（A3v2: 無法區分"格式效應"和"宣告式格式附帶的結構資訊效應"）

**處理**: 四項門禁任一不通過→不執行實驗。A3v2 因四門全 FAIL 被截停——這是成功的設計審查，不是失敗的實驗。
