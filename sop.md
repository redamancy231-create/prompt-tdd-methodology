# 对照实验设计 SOP (Prompt Experiment Design SOP)

> **版本**: v1.0.0（从 prompt-tdd 项目提炼，泛化为通用协议）  
> **来源**: A2+A3 实验的实战方法论，经 17+ 轮异后端审查验证  
> **生成模型**: DeepSeek-V4-Pro (via Claude Code CLI) · 2026-07-01

---

## §1 实验前检查清单 (CK1-CK6)

### CK1-CK3: Tier 1 硬门（缺一不可执行 Tier 1）

| 编号 | 检查项 | 通过标准 |
|------|--------|---------|
| **CK1** | 假设可证伪 | H0/H1 明确陈述 + 拒绝域事前定义 + 统计检验方法指定 |
| **CK2** | 因变量可操作化 | DV 有明确定义 + 测量协议可复制 + 评分 rubric 冻结 |
| **CK3** | 样本量充足 | 功效分析完成 + 最小兴趣效应量声明 + n ≥ min_n_per_arm |

### CK4-CK6: 条件触发（依据实验复杂度选择性执行）

| 编号 | 检查项 | 触发条件 | 通过标准 |
|------|--------|---------|---------|
| **CK4** | INVENTORY 等价性 | 双组 prompt 内容不同时 | 两组 prompt 产出清单逐项核对 + hash 冻结，防 IV 污染 |
| **CK5** | Train/Test 分离 | 需要迭代改进 prompt 时 | test 集在实验结束前不被用于 prompt 选择或参数调整 |
| **CK6** | 天花板探测 | DV 为 0/1 或比例指标时 | 嵌入反编造/极端测试用例评估检测空间 |

## §2 实验管线：Tier 0 → Tier 1

### 2.1 Tier 0: 描述性统计（工具链验证）

**目的**: 在投入大样本前验证数据收集和评分管线可行。

| 步骤 | 操作 | 成功判据 |
|------|------|---------|
| 1. 小样本收集 | n=4-8/臂，手动审核每条原始输出 | 无数据收集异常（截断/超时/格式错误） |
| 2. 评分管线验证 | 运行 score 脚本，检查 manifest 完整性 | 评分脚本运行正确，无缺失分 |
| 3. 描述性统计 | `analyze_experiment.py --tier 0` | 方向性描述 + 效应量粗略估计 |
| 4. 工程门判决 | 判断管线是否可进入 Tier 1 | PASS → 进入 Tier 1; FAIL → 修复后重跑 |

**Tier 0 不升级成熟度**。它只验证"管线能用"，不提供推断结论。

### 2.2 Tier 1: 推断统计（假设检验）

**前置条件**: Tier 0 工程门 PASS + CK1-CK3 全部通过。

| 步骤 | 操作 | 关键约束 |
|------|------|---------|
| 1. 大样本收集 | n ≥ 功效分析指定值 | raw_outputs/ 只增不覆盖 |
| 2. 双 LLM 异后端盲评 | 评分者 A（作者模型）+ 评分者 B（不同后端） | 评分者 B 不接触 A 的评分、作者自评、材料来源 |
| 3. 解盲 + 合并 | 合并 A/B 评分者，对齐 case_id | 评分前不告知分组 |
| 4. 统计推断 | `analyze_experiment.py --tier 1 --test-set test_set.json` | Wilcoxon + bootstrap CI + Cohen's κ |
| 5. 科学门判决 | 统计显著 + 效应量 ≥ 最小兴趣阈值 + 方向一致 | 三项全 PASS → 支持 H1; 任一 FAIL → 不支持 |

## §3 工程门 vs 科学门

对照实验有两道独立门——混淆两者是实验设计最常见的错误。

### 工程门（Engineering Gate）

**判什么**: 工具链是否可用？  
**通过标准**: 数据收集无异常 + 评分脚本运行正确 + manifest 完整性校验通过  
**失败后**: 修复工具链 → 重新收集（工程门失败不影响科学结论——这是管道问题，不是假设问题）

### 科学门（Science Gate）

**判什么**: 假设是否被支持？  
**通过标准**: 统计显著 + 效应量 ≥ 最小兴趣阈值 + 方向一致 + 评分者一致性达标（Cohen's κ ≥ 0.4 或 discordant 率 < 25%）（四项 **全部**）  
**失败后**: 诚实记录——阴性结果 ≠ 实验失败。A2/A3 科学门全部 FAIL，但仍然产出了有价值的阴性证据和方法论资产

> **关键教训**：**工程门 PASS + 科学门 FAIL = 有效阴性证据**。成熟度标注为 [E-]（ceiling-limited）而非 [E]，但仍然是可发表的诚实结论。

## §4 预注册锁 (.lock)

### 4.1 为什么需要

防止事后调整假设（HARKing——Hypothesizing After Results are Known）。

### 4.2 锁内容

```yaml
hypothesis: "<H0/H1 声明>"
analysis_plan:
  test_method: "Wilcoxon signed-rank | sign test | ..."
  effect_size_threshold: <最小兴趣效应量>
  stopping_rule: "<停止规则>"
test_set_sha256: "<SHA256 hash of test_set.json>"
lock_date: "<ISO 8601>"
```

### 4.3 操作

1. 在实验执行前，创建 `.lock` 文件
2. `git commit` 后不可修改（任何修改都会产生新 commit hash，可追溯）
3. 分析脚本自动校验 test_set SHA256——不匹配则 hard fail

## §5 INVENTORY 等价协议

### 5.1 问题

对照实验的双组 prompt 内容不同——但如何保证**信息量等价**而非**格式效应**？

### 5.2 协议

1. 分别运行 A 和 B 组 prompt 在相同输入上
2. 提取两组 prompt 产出的**检查项清单**（inventory）
3. 逐项核对两组 inventory 是否覆盖相同的检查项
4. 对 inventory 做 hash 冻结
5. 如发现不等价 → 修正 prompt 直到等价 → 重新冻结

> **A2 教训**：INVENTORY 等价性差异导致 IV（prompt 格式）与"信息量差异"混淆——B 组实际包含了 A 组没有的检查维度。

## §6 成熟度标注体系

| 标签 | 含义 | 升级条件 |
|------|------|---------|
| `[Sp]` | Specification — 设计草案 | 初始状态 |
| `[E]` | Experiment — 实验执行 | Tier 1 完成 + 科学门 PASS + 异后端审查闭合 |
| `[E-]` | Experiment — ceiling-limited | Tier 1 完成 + 科学门 FAIL（阴性或天花板） |
| `[F·1域]` | Framework — 单域集成 | [E] 结论写回框架，标注任务域限制 |
| `[J]` | Journal — 学术发表 | 经同行评审 |

**约束**：单域实验最多升级到 `[F·1域]`。跨域推广需 ≥2 域实验支持。

## §7 审查闭合协议

### 7.1 强制条件

- 异后端独立审查 ≥1 轮后方可宣布闭合
- 审查者不得与实验执行者同后端
- 审查者须在裸环境中审阅（不接触实验执行记录、作者自评、中间版本）

### 7.2 审查焦点

| 审查维度 | 检查内容 |
|---------|---------|
| 设计合理 | CK1-CK6 全部通过？ |
| 执行正确 | 工程门 PASS？预注册锁有效？ |
| 分析正确 | 统计方法适当？结论不超出效应量范围？ |
| 诚实记录 | 阴性结果如实报告？局限声明充分？ |

## §8 常见失败模式

### 8.1 天花板效应
**现象**: 模型接近完美 → 效应量不可检测  
**检测**: CK6 天花板探测（嵌入反编造/极端测试用例）  
**处理**: 诚实标注 [E-] ceiling-limited —— 这是有效结论，不是失败

### 8.2 IV 污染
**现象**: 对照组的 prompt 信息量不等价 → 检测到的是信息量效应，不是格式效应  
**检测**: CK4 INVENTORY 等价协议  
**处理**: 修正 prompt 直到等价 → 重新冻结

### 8.3 评分者分歧
**现象**: 两个 LLM 评分者对同一 case 给出不同评分  
**检测**: Cohen's κ < 0.4 或 discordant 率高  
**处理**: 采用保守评分策略（取较低分）；已知局限——LLM-LLM 评分 ≠ 人类评分准确性

### 8.4 HARKing
**现象**: 看到结果后调整假设  
**检测**: .lock 文件的 git commit 时间戳在实验执行之前  
**处理**: 预注册锁 + 科学门四项全 PASS 才能声明"支持 H1"

### 8.5 DV 退化（A3v2 教训）
**现象**: 实验迭代中 DV 选择逐步退化——从最初的构造效度关切，到后续版本中退化为"能拿到数据就行"  
**检测**（A3v2 审查发现的四个设计门禁，Codex + Qwen 双后端共识）：
1. **构造效度门**：DV 是否真正测量了假设指定的构念？（A3v2: accuracy 退化为0/1，丢失了路由决策的"质量"维度）
2. **格式×数量混淆门**：A/B 组 prompt 的格式差异是否与信息量差异正交？（A3v2: 声明式格式天然产生更多结构化字段，格式和信息量不可分）
3. **天花板逃脱门**：新的实验设计是否创造了比上一轮更大的检测空间？（A3v2: DV 退化后检测空间反而更小）
4. **效应量可解释门**：如果检测到 Δ，它是什么意思？（A3v2: 无法区分"格式效应"和"声明式格式附带的结构信息效应"）

**处理**: 四项门禁任一不通过→不执行实验。A3v2 因四门全 FAIL 被截停——这是成功的设计审查，不是失败的实验。
